# Debug Report: PRJ-394 — Patrol timestamps hallucinated, heartbeat missing

## Status
- **Phase:** D5 (Handoff — fix implemented)
- **Tier:** Config bug
- **Started:** 2026-04-04 18:55 CT
- **Root Cause Proven:** true

## Symptoms
1. Patrol Telegram reports show wrong timestamps (e.g. "1:30 AM CT" when actual time is 6:16 PM, "9:30 PM CT" at 6:00 PM)
2. Patrol appears dead after ~1 hour — no Telegram messages despite launchd job running fine

## IS / IS NOT
| Dimension | IS (observed) | IS NOT (not observed) |
|-----------|--------------|----------------------|
| What | Wrong timestamps in Claude's reports + missing Telegrams | Patrol actually stopped or crashed |
| Where | Claude's generated output (not the bash hour guard) | Bash script time logic (uses correct TZ=America/Chicago) |
| When | After feed went stale (~5:50 PM CT) — Claude stopped sending Telegrams for 0-new-listing cycles | Before 5:50 PM — earlier patrols did send Telegrams |
| Extent | Cosmetic (timestamps) + operational (no heartbeat) | No impact on scoring, tracking, or auth |

## Root Cause

**Three-element proof:**

1. **Reproduction:** Patrol log shows run at 18:16 reporting "1:30 AM CT", run at 17:29 reporting "10:30 PM CT". Claude has no reliable time source — the prompt doesn't include one.

2. **Isolation:** The bash script's hour guard uses `TZ=America/Chicago date +%H` correctly (returns 18, within 7-23). Only Claude's generated text has wrong times — it's hallucinating, not reading system time.

3. **Completeness:** Claude LLMs don't have access to system clocks. Without an injected timestamp, Claude guesses based on training data patterns. The heartbeat issue is separate: the prompt's Step 7 says "return a report" but doesn't mandate a Telegram for 0-new-listing cycles, so Claude sometimes skips it.
