Uplink daemon error log filled with 409 Conflict errors:
```
telegram.error.Conflict: can't use getUpdates method while webhook is active;
use deleteWebhook to delete the webhook first
```
All Telegram services down — no approvals, no relay, no notifications.

## Reproduction
```
$ grep 409 ~/.local/share/uplink/daemon.err | head -2

2026-04-04 16:47:25,024 [telegram.ext.Updater] ERROR: Exception happened while polling for updates.
```
Daemon polling every ~30s, all returning 409. Patrol log shows runs at 16:43, 16:58, 17:06, 17:21.

## Root Cause

**Three-element proof:**

1. **Reproduction:** 409 errors began at 16:47:25, immediately after the 16:43 patrol cycle. The patrol had `Bash(curl:*)` allowing ANY curl command, including `setWebhook`.

2. **Isolation:** The patrol Claude is the only new actor with unrestricted curl access to the bot token. HandoffMD has no `setWebhook` call in its codebase. No other system changed.

3. **Completeness:** Telegram only allows one mode per bot — long polling OR webhooks. Once `setWebhook` was called (by the patrol), the daemon's `getUpdates` calls all returned 409. Calling `deleteWebhook` restored service.

## ROOT-CAUSE-PROVEN: true

## Fix Approach

Replace `Bash(curl:*)` and `Bash(security:*)` with a pinned helper script (`bin/scout-telegram.sh`) that only supports `sendMessage` and `sendDocument`. The patrol Claude cannot call any other Telegram API method or access the Keychain directly. The `--allowedTools` pattern is pinned to the absolute path to prevent shim creation in temp dirs (Codex finding).
