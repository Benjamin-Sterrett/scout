# Debug Report: PRJ-383 — Headless patrol auth failure

## IS / IS NOT Matrix

| Dimension | IS | IS NOT |
|-----------|-----|--------|
| What | "Not logged in" error on every patrol run | A Chrome or MCP connection issue |
| Where | `bin/scout-patrol.sh` line 42, `--bare` flag | launchd plist, patrol prompt, Chrome extension |
| When | Every run since first 7 AM cycle on 2026-04-04 | Intermittent or time-dependent |
| Extent | 100% of runs (38 consecutive failures) | Partial failures or degraded behavior |

## Symptoms/Repro

**Symptom:** Headless patrol logs "Not logged in · Please run /login" on every cycle since 7 AM Apr 4.
**Repro:** Check `~/Projects/scout/data/logs/patrol-2026-04-04.log` — 38 consecutive identical failures from 07:01 to 11:16, every 15 minutes. Hour-guard and Chrome-check pass; failure occurs at the `claude` CLI invocation.

## Observations

- `launchctl list | grep scout` shows PID column = 1, confirming launchd job is loaded and running
- Pre-7 AM entries correctly skip with "Outside patrol hours" — script logic works
- No launchd-stderr.log output — script itself runs without errors
- `claude --bare` strips all user config per docs: "starts without loading any project context or user configuration"
- Auth tokens are stored in user config; `--bare` bypasses them entirely
- The `--bare` flag was added on Apr 3 to avoid MCP config loading issues, but it also blocks auth

## Hypotheses

1. **`--bare` flag strips auth context** — claude CLI in `--bare` mode skips loading user config/auth, causing "Not logged in"
2. Chrome extension not connecting — MCP server unavailable
3. launchd environment missing auth tokens

## Root Cause

**Hypothesis 1 confirmed.**

### Three-Element Proof

1. **Reproduction:** Every patrol run since 7 AM logs identical "Not logged in · Please run /login" — 38/38 failures. The hour-guard (pre-7 AM skips) and Chrome-check both pass, proving the script reaches the claude invocation.

2. **Isolation:** The `--bare` flag was added in the Apr 3 session specifically to avoid loading MCP configs. The `claude --help` docs confirm `--bare` "starts without loading any project context or user configuration." Auth is part of user configuration. Removing `--bare` while keeping `--print --chrome --permission-mode bypassPermissions` restores auth without changing other behavior.

3. **Completeness:** No other flag in the invocation affects auth. The launchd plist correctly sets `HOME` and `PATH` (verified by reading the plist). Chrome is running (the Chrome-check guard passes). The issue is exclusively the `--bare` flag stripping auth.

## ROOT-CAUSE-PROVEN: true

## Fix Approach

Remove `--bare` from line 42 of `bin/scout-patrol.sh`. Keep all other flags. Reload launchd plist after fix.
