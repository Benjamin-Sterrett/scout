# Debug Report: PRJ-388 — Patrol prompt not passed to CLI

## Symptoms/Repro

**Symptom:** Patrol log at 12:31 shows: "Error: Input must be provided either through stdin or as a prompt argument when using --print"
**Repro:** Check `~/Projects/scout/data/logs/patrol-2026-04-04.log` — 12:31 entry shows the error immediately after PRJ-386 merge.

## Observations

- Error appeared on the first patrol cycle after PRJ-386 added `--add-dir` and `--setting-sources` flags
- `--add-dir <directories...>` is variadic (Commander.js `...` syntax) — consumes subsequent non-flag args
- `--allowedTools <tools...>` is also variadic
- `$PROMPT` is passed as the last positional argument — after both variadic flags
- One of these variadic flags is consuming `$PROMPT` as its own argument, leaving no prompt for `--print`
- The prompt variable IS populated (read from file before cd /tmp, absolute path)

## Hypotheses

1. **`--add-dir` or `--allowedTools` variadic flags consume `$PROMPT`** — Commander.js variadic options eat subsequent positional args

## Root Cause

**Hypothesis 1 confirmed.** The trailing `$PROMPT` positional argument is consumed by a variadic flag, not passed as the CLI prompt.

### Three-Element Proof

1. **Reproduction:** Error appeared on first cycle after adding `--add-dir` (variadic flag). Previous version without `--add-dir` passed prompt successfully.
2. **Isolation:** The `--add-dir <directories...>` flag accepts multiple args. When followed by `--allowedTools "..."` (a flag), it stops. But the trailing `$PROMPT` is not a flag — it gets consumed.
3. **Completeness:** Fix is to pipe prompt via stdin instead of positional arg, which is unambiguous.

## Verification

- Script syntax check after fix
- Next patrol cycle log shows actual activity, not "Input must be provided" error

## ROOT-CAUSE-PROVEN: true

## Fix Approach

Change `"$PROMPT"` positional arg to `<<< "$PROMPT"` here-string stdin delivery.
