## Data Contracts (explicit interfaces between waves)

Each wave produces a typed contract that the next wave consumes. No wave reaches into another's internals.

```
Wave 1 produces:  RawListing          → unprocessed input (platform, raw_text)
Wave 2 produces:  ParsedListing       → structured fields + parse_confidence (high/medium/low)
Wave 1 produces:  ScoredListing       → ParsedListing + gate_results + dimension_scores + total_score + sla
Wave 3 produces:  ApprovedListing     → ScoredListing + approval (human gate, v1)
Wave 4 produces:  NormalizedTicket    → ApprovedListing → Linear ticket payload
```

### Parse Confidence (Wave 2, from day one)

Not just parsed/failed. Three levels:

| Confidence | Definition | Pipeline Behavior |
|------------|------------|-------------------|
| **high** | All key fields extracted (budget, tech stack, client history, clear description) | Score normally |
| **medium** | Some fields extracted, some inferred or missing (e.g., budget range guessed from description) | Score with warning flag, lower weight on inferred fields |
| **low** | Minimal extraction — mostly raw text, couldn't identify platform or structure | Present to user with "manual review needed" tag, don't auto-score |

This matters because the scoring engine needs to know whether it's working with clean data or guesses. A Clarity score of 4 means nothing if the parser couldn't actually extract the description properly.

## v1 Manual Flow (literal step-by-step)

```
1. Benjamin pastes listing text into CLI (stdin or --file)
2. CLI auto-detects platform if possible, or uses --platform flag
3. Parser extracts structured fields → ParsedListing with confidence level
4. Gates run → pass/fail per gate, rejected listings shown with reasons
5. Scoring runs on gate-passing listings → ranked by total score
6. CLI outputs: rank, score, SLA, dimension breakdown, bonuses, confidence
7. Benjamin reviews top N, picks one (or none)
8. Benjamin runs: python -m scout normalize <job_id> → creates Linear ticket
9. Dev-workflow takes over from the Linear ticket
```

The key: steps 7-8 are human gates. No auto-normalization in v1. Benjamin sees the scores, makes the call, then explicitly triggers ticket creation.
